# План практики: фронтенд с ИИ (React + TypeScript)

**Формат:** 4 недели · 5 рабочих дней в неделю · 4 часа в день · **80 часов** · 20 рабочих дней
**Проект:** «Mini-Shop» — витрина интернет-магазина (данные из публичного API)
**Режим:** работа с ИИ-инструментами (Cursor, Copilot, ChatGPT, Claude и т.п.)

---

## Как работать

**Инструменты и данные**
- ИИ-инструменты разрешены и приветствуются. Бэкенд писать не нужно — данные берём из **[DummyJSON](https://dummyjson.com/)** (товары, поиск `/products/search?q=`, категории, пагинация `limit`/`skip`).
- Стек: **Vite + React 18 + TypeScript (strict) + React Router**. Состояние — Context API или Zustand.
- **Дизайн-система (обязательно):** одна из готовых UI-библиотек на выбор — **Material UI (MUI), Ant Design, Chakra UI или Mantine**. Весь интерфейс собирается на её компонентах; кастомные стили — через штатный механизм библиотеки (тема, токены, `sx`/styled). Своя CSS — только точечно, где библиотеки не хватает. Одну систему не смешивать с другой; тему обязательно настроить под проект, а не оставлять дефолтной.

**Главное правило ИИ-режима**
> Ты отвечаешь за весь код. ИИ — помощник, а не автор. Любой кусок кода нужно **прочитать, понять и уметь защитить**. Вставка без понимания не засчитывается.

**Ежедневная рутина (каждый день)**
1. В начале дня — посмотреть цель и задачи дня.
2. Перед каждым коммитом — **прочитать и проверить** код, который сгенерировал ИИ (нет ли лишнего, ошибок, дублирования).
3. По ходу — короткие заметки в `AI_REPORT.md`: удачные промпты и случаи, где ИИ ошибся.
4. В конце дня — осмысленный коммит в Git.

**Два обязательных артефакта в репозитории**
- `README.md` — как запустить, стек, ключевые решения.
- `AI_REPORT.md` — как использовался ИИ (заполняется всю практику, финал — на 19-й день).

**Если отстаёшь от графика:** в первую очередь доделывай базовый функционал (каталог → корзина → заказ). Дополнительные фичи (избранное, тёмная тема) — то, чем можно пожертвовать или что можно заменить.

## Неделя 1 — Старт, каркас, каталог

### День 1 — Подготовка и настройка
- Настроить ИИ-инструменты. Создать `AI_REPORT.md` и начать вести лог.
- Инициализировать проект: Vite + React + TS (strict).
- **Настроить ESLint + Prettier:** правила для React и TypeScript, интеграция Prettier с ESLint (без конфликтов), npm-скрипты `lint` и `format`, форматирование при сохранении. Код обязан проходить линтер без ошибок и предупреждений.
- *(Опционально)* Husky + lint-staged: автопроверка ESLint/Prettier перед коммитом, чтобы в репозиторий не попадал неотформатированный код.
- Выбрать и установить дизайн-систему (MUI / Ant Design / Chakra / Mantine), подключить её провайдер темы.
- Создать репозиторий на GitHub, первый коммит, продумать структуру папок.
- Изучить API: сделать тестовый запрос, посмотреть формат данных.
- **Готово, когда:** проект запускается (`npm run dev`), ESLint и Prettier настроены и проходят без ошибок, дизайн-система подключена, репозиторий создан, `AI_REPORT.md` начат.

### День 2 — Каркас и маршрутизация
- Настроить React Router: маршруты `/` (каталог), `/product/:id` (товар), `/cart` (корзина).
- Настроить тему дизайн-системы под проект (основной цвет, типографика) — не оставлять полностью дефолтной.
- Собрать общий layout на компонентах библиотеки: шапка с навигацией и иконкой корзины, контейнер контента, подвал.
- Создать заглушки страниц и описать типы данных (`Product` и др.) по ответу API.
- **Готово, когда:** переходы между страницами работают, layout собран на компонентах дизайн-системы с настроенной темой.

### День 3 — Загрузка каталога
- Сделать слой работы с API (функция запроса товаров) с обработкой загрузки и ошибки.
- Вынести запрос в кастомный хук (например, `useProducts`).
- Отрендерить сетку карточек: картинка, название, цена, рейтинг.
- **Готово, когда:** на главной видны реальные товары, корректно показываются состояния загрузки и ошибки.

### День 4 — Карточка товара и стили каталога
- Компонент `ProductCard` на основе карточки из дизайн-системы; сетка на её Grid-компонентах, адаптив (mobile-first).
- Состояние «ничего не найдено».
- **Готово, когда:** каталог выглядит опрятно и адаптивно, оформлен в стиле выбранной дизайн-системы.

### День 5 — Страница товара + ревью недели
- Детальная страница: запрос товара по `id`, полная информация о товаре.
- Кнопка «В корзину» (пока заглушка).
- Ревью своего кода и рефакторинг, обновить `README.md`.
- Записать в `AI_REPORT.md` итоги недели: 1–2 удачных промпта и случаи ошибок ИИ.
- **Готово (конец недели 1):** работают каталог и страница товара, код чистый, ведётся AI-лог.

## Неделя 2 — Поиск, фильтры, состояние корзины

### День 6 — Поиск
- Поле поиска с задержкой запроса (debounce).
- Корректная обработка пустого результата и загрузки при поиске.
- **Готово, когда:** поиск работает плавно, без лишних запросов на каждый символ.

### День 7 — Фильтры и сортировка
- Фильтр по категориям (получить список категорий из API).
- Сортировка по цене и по рейтингу.
- **Готово, когда:** можно фильтровать и сортировать каталог, состояния не конфликтуют с поиском.

### День 8 — Пагинация
- Пагинация или бесконечная прокрутка (`limit`/`skip`).
- **Готово, когда:** каталог листается, подгрузка работает корректно.

### День 9 — Состояние корзины
- Глобальное состояние корзины (Context или Zustand): добавить, удалить, изменить количество, посчитать сумму.
- Подключить кнопку «В корзину» на карточке и детальной странице.
- **Готово, когда:** товары реально добавляются в корзину, сумма считается верно.

### День 10 — Персист корзины + ревью недели
- Сохранение корзины в `localStorage` (не теряется при перезагрузке).
- Бейдж с количеством товаров в шапке.
- Ревью кода, обновить `AI_REPORT.md`.
- **Готово (конец недели 2):** корзина работает и сохраняется между перезагрузками.

## Неделя 3 — Оформление заказа и доп. фичи

### День 11 — Страница корзины
- Полноценная страница корзины: список товаров, изменение количества, удаление, итоговая сумма.
- Состояние пустой корзины.
- **Готово, когда:** корзиной можно полностью управлять с её страницы.

### День 12 — Форма заказа (поля)
- Форма оформления на компонентах дизайн-системы (поля ввода, select и т.п.): имя, email, адрес, телефон.
- Базовое управление формой и состоянием полей.
- **Готово, когда:** форма отображается на компонентах библиотеки, поля управляемые.

### День 13 — Валидация и подтверждение заказа
- Полная валидация (обязательные поля, формат email, ошибки под полями, блокировка кнопки при невалидных данных).
- Экран подтверждения заказа и очистка корзины после оформления.
- **Готово, когда:** работает полный путь «каталог → корзина → заказ → подтверждение».

### День 14 — Доп. фича 1: «Избранное»
- Добавление/удаление из избранного, отдельный раздел, сохранение в `localStorage`.
- **Готово, когда:** избранное работает и сохраняется.

### День 15 — Доп. фича 2: тёмная тема + ревью недели
- Переключатель светлой/тёмной темы средствами дизайн-системы (её механизм тем) с сохранением выбора.
- Ревью кода, обновить `AI_REPORT.md`.
- **Готово (конец недели 3):** продукт функционально готов, реализованы 2 доп. фичи.

## Неделя 4 — Качество, полировка, защита

### День 16 — Доступность и финальный адаптив
- Семантическая разметка, `alt` у картинок, видимый фокус, навигация с клавиатуры.
- Проверить весь интерфейс на мобильном и десктопе, поправить недочёты.
- **Готово, когда:** интерфейс корректен на разных экранах и доступен с клавиатуры.

### День 17 — Небольшая фича: уведомления и скелетоны
- Всплывающие уведомления (toast) при добавлении в корзину и в избранное.
- Скелетоны загрузки вместо пустого экрана во время ожидания данных.
- **Готово, когда:** действия сопровождаются уведомлениями, при загрузке видны скелетоны.

### День 18 — Производительность и чистка кода
- Мемоизация где уместно, ленивая загрузка изображений, разделение кода по маршрутам (lazy + Suspense).
- Финальная чистка: убрать мусор, привести стиль кода к единому виду.
- **Готово, когда:** нет лишних перерисовок/мусора, проект собирается без предупреждений линтера.

### День 19 — Полировка UI/UX + README + AI-отчёт
- Полировка интерфейса: единообразное использование темы и компонентов дизайн-системы, аккуратные отступы, состояния hover/active, плавные переходы, пустые состояния и обработка ошибок.
- Дописать `README.md` (запуск, стек, ключевые решения, скриншоты).
- Финализировать `AI_REPORT.md`: лучшие промпты, 2–3 случая ошибок ИИ и как их поймали, где ИИ помог, а где мешал, короткая рефлексия.
- **Готово, когда:** интерфейс выглядит цельно, оба артефакта готовы.

### День 20 — Демо и защита
